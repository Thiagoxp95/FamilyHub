# Mac Release and Auto-Update Design

## Context

FamilyHub is a personal Electron app that will be developed on one Mac and run on an Apple Silicon kitchen Mac. The goal is to tag releases from the development machine, let GitHub Actions package the app, publish a public GitHub Release, and let the kitchen Mac install updates from that release feed.

The current Electron app already uses `electron-builder` and has a macOS `dmg` target. There is no GitHub Actions workflow, no release publishing setup, and no updater IPC/UI yet.

## Goals

- Package only macOS Apple Silicon builds.
- Publish a downloadable `.dmg` to public GitHub Releases.
- Support automatic update checks from the installed app.
- Keep the release system simple enough for a single personal device.
- Avoid embedding GitHub credentials in the app.

## Non-Goals

- Windows or Linux packaging.
- Intel macOS builds.
- Private update feeds.
- Custom update hosting such as S3, R2, or a database-backed release service.
- In-app release notes or complex staged rollout controls.

## Release Architecture

Use GitHub Releases as the update provider and `electron-updater` in the app. Since releases are public, the installed app can read update metadata without a token.

`electron-builder` should build macOS ARM64 artifacts with both `dmg` and `zip` targets. The DMG is the human-friendly installer. The ZIP and generated `latest-mac.yml` metadata are required for the macOS auto-update path used by `electron-updater`.

The Electron package should declare GitHub publishing configuration. The workflow can rely on the repository context and `GITHUB_TOKEN`, but the config should still be explicit enough that local and CI release behavior are easy to understand.

## GitHub Actions Flow

Create `.github/workflows/release-mac.yml`.

The workflow runs on version tags such as `v0.1.0` and uses a macOS runner. It installs npm dependencies, runs the Electron workspace checks, builds only `mac-arm64`, and publishes artifacts to the GitHub Release.

Expected published assets:

- `FamilyHub-<version>-arm64.dmg`
- `FamilyHub-<version>-arm64-mac.zip`
- DMG/ZIP blockmap files generated by electron-builder
- `latest-mac.yml`

The workflow should fail release builds if signing inputs required for notarized releases are missing. That prevents accidentally publishing artifacts that cannot update reliably on the kitchen Mac.

## Signing and Notarization

macOS auto-update expects signed applications, and normal macOS distribution works best when the app is also notarized. Because this is for personal use, the workflow can be minimal, but it should still support the standard Apple Developer signing path.

Required GitHub secrets:

- `MAC_CERTIFICATE`: base64-encoded Developer ID Application certificate export.
- `MAC_CERTIFICATE_PASSWORD`: password for the exported certificate.
- `APPLE_ID`: Apple Developer account email.
- `APPLE_APP_SPECIFIC_PASSWORD`: app-specific password for notarization.
- `APPLE_TEAM_ID`: Apple Developer team ID.

The implementation should document how to create these secrets. Local development builds do not need signing.

## App Update Flow

The main process owns all updater behavior. Renderer code talks to it only through the preload bridge.

Updater behavior:

- Do nothing in dev mode.
- Check once after the app is ready.
- Check periodically while the app is open.
- Download an available update automatically.
- Broadcast updater state to all renderer windows.
- Install only when the renderer calls an explicit restart action.

Updater states:

- `idle`
- `checking`
- `available`
- `downloading`
- `downloaded`
- `not-available`
- `error`

The preload bridge exposes a small `window.familyHub.updater` API:

- `getStatus()`
- `check()`
- `install()`
- `onStatus(callback)`

The renderer displays a compact update control in the existing header area. It should stay quiet when there is no update. Once an update has downloaded, it shows a clear restart action.

## Error Handling

Updater errors should not interrupt the kitchen dashboard during background checks. Errors should be stored in updater state and shown only when the user is interacting with update controls or when an update was expected but failed.

Release workflow failures should be explicit. Missing signing secrets, missing update metadata, or missing release artifacts should fail the workflow.

## Testing and Verification

Unit tests should cover updater state mapping and IPC behavior where practical. Renderer tests should cover the visibility and labels for the update control states if the component is extracted.

Manual verification:

- Run `npm run typecheck`.
- Run `npm run lint`.
- Run `npm --workspace @family-hub/electron run build`.
- Run the release workflow from a version tag.
- Confirm the GitHub Release contains `.dmg`, `.zip`, blockmaps, and `latest-mac.yml`.
- Install an older signed build on the kitchen Mac, publish a newer tag, and confirm the app downloads and restarts into the new version.

## Open Decisions Resolved

- Update provider: public GitHub Releases.
- Platform: macOS Apple Silicon only.
- Release trigger: version tags.
- Signing: include Apple Developer signing/notarization path, even for personal use.
